## 2. 变更1：会议费两阶段审核流程

### 2.1 新状态机

```
                    ┌──────────────────────────────────────────────────────┐
                    │              两阶段审核状态机（新）                     │
                    │                                                      │
                    │  unpaid ──[上传凭证]──→ voucher_submitted             │
                    │                           │                          │
                    │                      [财务初审]                        │
                    │                   ┌──────┴──────┐                    │
                    │                   │             │                    │
                    │              初审通过        初审驳回                   │
                    │                   │             │                    │
                    │                   ▼             ▼                    │
                    │          invoice_pending    unpaid                   │
                    │          (可填参会信息)     (重新上传凭证)              │
                    │                   │                                  │
                    │           [上传发票 + OCR比对]                         │
                    │                   │                                  │
                    │              [财务终审]                               │
                    │           ┌──────┴──────┐                            │
                    │           │             │                            │
                    │      终审通过        终审驳回                          │
                    │           │             │                            │
                    │           ▼             ▼                            │
                    │      confirmed    invoice_pending                    │
                    │      (报名完成)   (重新上传发票)                        │
                    └──────────────────────────────────────────────────────┘
```

### 2.2 ConferenceReg 数据模型变更

```typescript
// 旧（当前原型）
interface ConferenceReg {
  status: "unpaid" | "pending" | "approved_unfilled" | "submitted";
  paymentVoucher?: string;
  invoice?: string;
  // ... form fields
}

// 新（两阶段审核）
interface ConferenceReg {
  status: "unpaid"              // 初始：未缴费
        | "voucher_submitted"   // 凭证已上传，待初审
        | "voucher_rejected"    // 凭证初审驳回
        | "invoice_pending"     // 初审通过，待上传发票（可填参会信息）
        | "invoice_submitted"   // 发票已上传，待终审
        | "invoice_rejected"    // 发票终审驳回
        | "confirmed";          // 终审通过，报名确认

  // 第一阶段
  paymentVoucher?: string;       // 转账凭证
  voucherSubmitTime?: string;    // 凭证提交时间
  voucherAuditTime?: string;     // 初审时间
  voucherRejectReason?: string;  // 初审驳回原因

  // 第二阶段
  invoiceUrl?: string;           // 发票文件
  invoiceSubmitTime?: string;    // 发票提交时间
  invoiceAuditTime?: string;     // 终审时间
  invoiceRejectReason?: string;  // 终审驳回原因
  invoiceDeadline?: string;      // 发票上传截止日（凭证初审通过 + 7工作日）
  invoiceExtendedDeadline?: string; // 手动延长期限

  // 参会信息（invoice_pending 状态时可填写）
  name: string;
  gender: "男" | "女";
  unit: string;
  role: "学生" | "教师" | "嘉宾";
  accommodation: "单间" | "双人间" | "自行安排";
  session: string;
  presentationType: "口头报告" | "展板报告" | "仅参会";
  reportTitle?: string;
  abstractFileName?: string;
  lastUpdated?: string;
}
```

### 2.3 交互流程详细设计

```
阶段一：凭证初审

  用户进入会议详情（status = unpaid）
    → 点击 [ 缴纳会议注册费 ]
    → 选择身份（学生/教师/嘉宾），系统计算金额
    → 展示收款账户信息
    → 线下转账
    → 上传转账凭证（JPG/PNG ≤5MB）
    → 提交 → status = "voucher_submitted"
    → toast "凭证已提交，财务初审中（通常1-3个工作日）"

  财务初审通过（后台操作）:
    → status = "invoice_pending"
    → 设置 invoiceDeadline = 初审通过日 + 7工作日
    → 通知用户 "凭证初审已通过，请于{deadline}前上传电子发票。
                   您现在可以填写参会信息。"
    → 解锁参会信息填写权限

  财务初审驳回（后台操作）:
    → status = "voucher_rejected"
    → 附带驳回原因
    → 通知用户重新上传凭证


阶段二：发票终审

  用户进入会议详情（status = invoice_pending）
    → 发票上传区域显示截止日倒计时
    → 可填写参会信息（与旧版 approved_unfilled 形态一致）
    → 上传电子发票（JPG/PNG/PDF ≤10MB）
    → 系统 OCR 识别 + 金额比对
      ├── 比对通过 → status = "invoice_submitted" → "发票已提交，财务终审中"
      └── 比对不通过 → toast "发票金额与凭证金额不一致，请检查后重新上传"

  财务终审通过（后台操作）:
    → status = "confirmed"
    → 通知用户 "报名已确认"

  财务终审驳回（后台操作）:
    → status = "invoice_rejected"
    → 附带驳回原因
    → 发票上传区重新开放


宽限期处理:

  发票上传截止日到达时:
    → 系统自动检查 status === "invoice_pending" 且 now > invoiceDeadline
    → status 变更为 "invoice_overdue"（新增状态）
    → 参会信息填写权限锁定（已填写的数据保留）
    → 通知用户 "发票上传已逾期，请尽快上传以完成报名确认"

  逾期后用户上传发票:
    → status = "invoice_submitted"
    → 进入终审流程

  特殊情况手动延期:
    → 财务后台可为订单设置 invoiceExtendedDeadline
    → 前端以 invoiceExtendedDeadline 为准
```

## 3. 变更2：发票必传 + 宽限期机制

### 3.1 适用范围

| 缴费类型 | 发票要求 | 宽限期 | 逾期后果 |
|----------|---------|--------|---------|
| 统一会员费 | 必传 | 初审通过后 7 个工作日 | 会员资格锁定，待上传+终审后解锁 |
| 会议注册费 | 必传 | 初审通过后 7 个工作日 | 报名资格锁定（见 §2.3 宽限期处理） |

### 3.2 会员费审核增加两阶段

此前会员费审核未明确区分阶段，现在与会议费对齐：

```
会员费状态机（新增两阶段）:

not_member
  → [转账 + 上传凭证] → voucher_submitted → [财务初审]
    ├── 初审通过 → invoice_pending（可绑定分会）
    │   ├── [上传发票] → invoice_submitted → [终审]
    │   │   ├── 通过 → active（正式会员）
    │   │   └── 驳回 → invoice_rejected → 重新上传发票
    │   └── [逾期] → invoice_overdue → 会员资格锁定
    └── 初审驳回 → voucher_rejected → 重新上传凭证
```

**关键变化：** 此前原型中会员费审核是一阶段（pending → active/rejected），现在改为两阶段，与会议费一致。

### 3.3 兜底方案

后台需提供：
- **手动延期：** 财务可为单个缴费记录设置 `invoiceExtendedDeadline`
- **手动解锁：** 总管理员可为逾期用户手动解锁（绕过宽限期限制）
- **延期理由记录：** 每次手动延期需填写理由，计入操作日志

## 4. 变更3：会员到期后会议资格分级处理

### 4.1 规则表

| 会员到期时的会议状态 | 处理方案 | 后续操作 |
|---------------------|---------|---------|
| 已报名 + **已终审通过** (`confirmed`) | **资格保留** | 会议正常参加，不受会员到期影响 |
| 已报名 + **待发票终审** (`invoice_submitted`) | **资格锁定** | 续费+终审通过后解锁 |
| 已报名 + **仅凭证初审** (`voucher_submitted` / `invoice_pending` / `invoice_overdue`) | **资格失效** | 需续费后重新按流程操作 |
| 未报名 + 任意 | 无影响 | 续费后可正常报名 |

### 4.2 实现逻辑（MembershipContext）

```typescript
// 会员到期处理函数（当检测到会员到期时调用）
function handleMembershipExpiry() {
  // 1. 遍历所有 conferenceRegs
  for (const [confId, reg] of Object.entries(conferenceRegs)) {
    switch (reg.status) {
      case "confirmed":
        // 保留不动
        break;
      case "invoice_submitted":
        // 锁定：标记为 frozen（新增标记字段）
        reg.frozenDueToExpiry = true;
        break;
      case "voucher_submitted":
      case "voucher_rejected":
      case "invoice_pending":
      case "invoice_overdue":
      case "invoice_rejected":
        // 自动失效
        delete conferenceRegs[confId];
        break;
    }
  }

  // 2. 清空分会绑定
  boundBranches = [];

  // 3. 更新会员状态
  societyMembership.status = "expired";
}
```

### 4.3 续费后的恢复逻辑

```typescript
// 会员续费 + 终审通过后
function handleMembershipRenewal() {
  // 1. 恢复被冻结的会议资格
  for (const [confId, reg] of Object.entries(conferenceRegs)) {
    if (reg.frozenDueToExpiry) {
      reg.frozenDueToExpiry = false;  // 解锁
    }
  }

  // 2. 用户需手动重新绑定分会
  // (不自动恢复绑定——用户可能改变了分会偏好)
}
```

## 5. 变更4：会员费金额后台可配置

### 5.1 配置模型

```typescript
// shared/constants.ts（前端默认值，可被后台配置覆盖）

export const MEMBERSHIP_FEE_CONFIG = {
  // 默认金额（后台未配置时使用）
  default: {
    standard: 200,    // 普通会员
    student: 100,     // 学生会员（预留）
    corporate: 5000,  // 单位会员（预留）
  },
  // 当前年度
  currentYear: 2026,
};

// 运行时从 API / localStorage 读取（原型阶段用 localStorage 模拟）
export function getMembershipFee(memberType: string = "standard"): number {
  // 原型：从 localStorage 读取配置，fallback 到默认值
  const stored = localStorage.getItem("paleo_membership_fee_config");
  if (stored) {
    const config = JSON.parse(stored);
    return config[memberType] || MEMBERSHIP_FEE_CONFIG.default[memberType];
  }
  return MEMBERSHIP_FEE_CONFIG.default[memberType];
}
```
