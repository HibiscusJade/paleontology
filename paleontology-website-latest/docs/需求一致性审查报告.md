# 需求一致性审查报告

> **审查时间：** 2026-06-10
> **审查范围：** 原型实际实现 vs `需求整理理解_V2.md` vs `中国古生物学会网站完整需求规格文档_V2.md`
>
> **结论：** 核心业务逻辑方向一致（统一会员体系、分会自由绑定、会议独立缴费），但存在 **2处重大流程差异**、**5处模糊/歧义需求**、**3处未实现的权限规则**，以及若干值得讨论的需求合理性问题。

---

## 一、重大流程差异（原型 vs 需求文档）

### 🔴 差异1：会议费缴费审核流程——两阶段审核 vs 一次性审核

这是最关键的差异。两份需求文档都描述了**两阶段审核流程**，但原型实现为**一次性审核**。

**需求文档描述（完整规格 §14.4，需求整理 §3.2）：**

```
第4步：上传缴费凭证
  → 智能识别后，状态变更为"初审通过：可填写参会信息"

第5步：上传电子发票
  → OCR识别+比对通过 → 进入人工审核

第6步：财务审核
  → 审核通过 → 获得参会资格
```

即流程为：
```
凭证上传 → 初审通过(可填参会信息) → 发票上传 → OCR比对 → 终审 → 获得参会资格
```

**原型实现（Services.tsx 第1068-1203行）：**

```
凭证 + 发票同时上传 → 提交 → pending → 审核通过 → approved_unfilled → 填写参会信息
```

即流程为：
```
凭证+发票一起上传 → 一次审核 → 可填参会信息
```

**差异要点：**

| 维度 | 需求文档 | 原型 |
|------|---------|------|
| 审核阶段 | 两次（初审+终审） | 一次 |
| 凭证上传后 | 可立即填参会信息 | 仍需等审核 |
| 发票上传时机 | 独立步骤，在凭证之后 | 与凭证同步骤 |
| 状态流转 | unpaid→凭证已传(可填参会)→发票已传→审核通过 | unpaid→pending→approved_unfilled→submitted |
| 发票是否必传 | 是（后续独立步骤） | 选传 |

**影响评估：** 需求文档的设计逻辑是"先让用户填写参会信息（凭证上传后立即开放），同时上传发票完成财务闭环"。原型设计更简单但意味着用户在审核通过之前不能填写参会信息。需确认哪种流程更符合学会财务审计要求。

---

### 🔴 差异2：会员费凭证与发票的关系——需求有OCR比对，原型无

**需求文档：**
- §14.2：凭证OCR识别 → 发票OCR识别 → **系统自动比对凭证金额与发票金额是否一致** → 比对不通过提示重新上传
- 这是一个智能审核的前置步骤

**原型实现：**
- 凭证和发票都是模拟点击即上传（实际产品中确实会做OCR，但当前无任何识别逻辑）
- 无比对步骤
- 发票为选传而非必传

**影响评估：** OCR+比对是需求文档的**核心智能审核特性**，原型阶段缺失是正常的。但**发票是否必传**这一点需要确认：如果发票必传，缴费流程至少需要有"发票催缴"状态。

## 二、原型未实现的需求规则

### 🟡 缺失1：会议列表未按已绑定分会过滤

**需求明确要求（完整规格 §6.2.2，需求整理 §3.1）：**

> "用户仅能看到已绑定分会发布的会议。未绑定分会的会议完全不可见。"

**原型现状：** 会议Tab展示全部9场会议，无绑定分会过滤逻辑。

**影响：** 这是核心权限隔离规则，原型阶段暂未实现。实际产品必须严格过滤。

---

### 🟡 缺失2：会员到期后自动解除所有分会绑定

**需求明确要求（完整规格 §14.3）：**

> "会员到期自动失效 → 所有分会绑定自动失效 → 续费后需重新绑定"

**原型现状：** 无自动解绑逻辑。`boundBranches` 存储在 localStorage 中，会员过期后数据依然保留，UI上虽有锁定但数据未被清除。

**建议：** 会员过期时清空 `boundBranches` 数组。

---

### 🟡 缺失3："加入会员"按钮的两个行为路径

**需求文档隐含要求：**

> "未登录 → 引导注册；已登录非会员 → 引导缴费"

**原型实现（PartyLayout.tsx 第199-215行）：**
```typescript
if (!isLoggedIn || societyMembership.status !== "active") {
  显示 "加入会员" 按钮
}
if (!isLoggedIn) → 注册弹窗
else → window.location.href = "/services?tab=member"  // 已登录非会员
```

原型中已登录非会员被跳转到会员服务Tab主页，而非**直接进入缴费流程**。这是一个微妙的UX差异：需求似乎期望点"加入会员"直接开始缴费，而原型多了一层引导。哪种更好取决于实际用户测试。

---

### 🟡 缺失4：过期状态的颜色

| 来源 | expired状态颜色 |
|------|----------------|
| 需求文档 §6.1.4 | **黄色**"已过期" |
| 原型 | **灰色**"会员已过期" |

建议统一为需求文档的黄色（更醒目，符合警示语义）。

## 三、模糊/歧义需求

### ⚠️ 模糊1：会议费审核驳回后是否回到"未缴费"状态？

**需求文档 §14.4 只说：**

> "审核驳回 → 填写原因，用户重新提交"

但未明确"重新提交"是指：A) 仅重新上传凭证/发票，还是 B) 需要重新走完整的缴费+上传流程？

**原型实现：** 选择B，驳回后 `conferenceReg.status` 回到 `"unpaid"`，用户需重新走整个缴费流程。

**建议：** 明确为B（保持单笔单次原则，避免一个会议产生多条缴费记录），并在需求文档中补充说明。

---

### ⚠️ 模糊2：会议状态（系统级）与用户报名状态（个人级）的术语混淆

需求文档 §6.2.4 描述了会议整体状态：

| 状态 | 触发条件 |
|------|----------|
| 报名中 | 会议发布后至截止日期前 |
| 已截止 | 超过缴费截止日期 |
| 进行中 | 会议日期范围内 |
| 已结束 | 会议日期过后 |

需求文档 §6.2.5 描述了用户卡片状态：

| 卡片状态 | 标签颜色 |
|----------|----------|
| 未缴费 | 默认 |
| 待审核 | 黄色 |
| 初审通过 | 蓝色 |
| 审核通过 | 绿色 |
| 已截止 | 灰色 |
| 已结束 | 灰色 |

**问题：** 这两组状态被放在一起而没有明确区分。一个用户的会议卡片上应该同时展示"会议整体状态"和"个人报名状态"两个维度。原型实际上通过不同层面的UI解决了这个问题（会议卡片显示会议状态，会议详情页显示个人报名状态），但需求文档中对这个分层不够清晰。

**建议：** 在需求文档中新增"状态分层说明"一节。

---

### ⚠️ 模糊3："即将开启"到"正在报名"的切换时机

需求文档有"即将开启"这个会议状态，原型中 conf-7（地球生物学前沿论坛）显示"即将开启"，但其缴费截止日期是 `2026-06-20`（实际上已过，因为当前日期是 2026-06-10）。

**问题：** "即将开启"到"正在报名"的切换是由什么触发的？是固定的发布时间？还是到达某个日期自动切换？

**建议：** 明确定义：
- 会议发布后、缴费未开始 → "预告通知"
- 缴费开放后、未截止 → "正在报名"
- 缴费截止后、会议未开始 → "报名已截止"
- 会议日期内 → "进行中"
- 会议结束后 → "已结束"

---

### ⚠️ 模糊4：摘要截止日期与缴费截止日期的关系

**需求描述：** 摘要修改截止日期 < 缴费截止日期（一般摘要截止更早）。

**边缘情况：** 如果用户在缴费截止日当天缴费，但摘要截止日早已过去，用户缴费后是否还能提交摘要？

**原型实现：** 两个截止日期独立校验。缴费看 `feeDeadline`，摘要上传看 `abstractDeadline`。如果摘要已截止，上传区域显示红色警告并隐藏上传按钮。

**建议：** 在需求文档中补充说明：缴费截止日期应早于或等于摘要截止日期。如果缴费截止日晚于摘要截止日，应在会议详情页显著提示用户。

---

### ⚠️ 模糊5："学生身份如何验证"

需求文档 §19.1 Q2 明确标记为**待确认**：

> "学生身份如何验证？是否需要上传学生证？还是仅靠用户自行声明？"

**原型实现：** 用户自行选择身份，无验证。这在实际产品中可能导致非学生用户选学生身份获取较低注册费。

**建议：** 至少增加"上传学生证照片"的可选步骤，或接受"线下会议签到时核验学生证"的方案。

---

## 四、需求合理性质疑

### 质疑1：线下转账 + 上传凭证的缴费方式过于繁琐

**现状：** 所有缴费（会员费+会议费）都要求：
1. 用户离开网站 → 去银行/网银转账
2. 截图保存凭证
3. 回到网站上传凭证
4. 等待人工审核（1-3个工作日）

**问题：**
- 每一步都是流失点
- 转账可能填错备注导致财务匹配困难
- 审核周期长，用户无法即时获得会员资格/参会资格

**需求文档 §19.4 自身也提出了此问题（S2：建议增加在线支付选项）。**

**建议：** 即使保留线下转账为主渠道（可能是财务制度要求），也应在缴费页面显著展示完整的操作指引，减少用户困惑。

---

### 质疑2：会员到期后自动解除所有分会绑定 + 续费后需重新绑定

**需求文档 §14.3：**

> "会员到期自动失效 → 所有分会绑定自动失效 → 续费后需重新绑定"

**问题：** 这会导致：
- 用户每年都要重新绑定一次分会（最多11个分会要逐一重新绑定）
- 如果用户在会员到期前报名了未来的会议，到期后绑定失效，会议资格如何处理？

**需求文档自身也标记了此问题（§19.1 Q1）：**

> "会员到期后，已绑定分会全部自动解除。已缴费但尚未举行的会议参会资格是否保留？"

**§19.4 S5 甚至建议了相反的规则：**

> "建议采用'缴费时锁定资格'规则——会员到期后不追溯取消已报名会议资格"

**建议：** 采用以下方案：
1. 会员到期后：分会绑定**暂不自动解除**，给30天宽限期
2. 宽限期内续费 → 绑定全部保留
3. 宽限期过后 → 自动解除所有绑定
4. 已缴费的会议资格**始终保留**（不受会员状态变化影响）

---

### 质疑3：会员费金额"统一标准"但未给出具体数字

两份需求文档都写"统一会员费金额（如每年XX元）"，未给出具体金额。原型硬编码为 ¥200/年。

**建议：** 确认具体金额，或至少给出建议范围（如 ¥100-300/年）。

---

## 五、两份需求文档之间的不一致

### 不一致1：需求整理无OCR细节，完整规格有

| 文档 | OCR描述 |
|------|--------|
| 需求整理_V2 | "凭证/发票先OCR识别，再人工审核"（一句话） |
| 完整规格_V2 | 详细描述了OCR识别字段（金额、收款方、付款方、日期）和比对逻辑 |

**建议：** 需求整理同步补充OCR详细说明。

---

### 不一致2：会议费审核流程在两份文档中详略不一致

| 文档 | 描述 |
|------|------|
| 需求整理_V2 §3.2 | "上传凭证 → 审核通过后（初审通过），状态变更为'可填写参会信息' → 上传发票 → 等待终审" |
| 完整规格_V2 §14.4 | 有完整的7步流程图和详细步骤说明 |

两份文档都描述了"两阶段审核"（凭证初审 → 发票终审），但需求整理更简略。**这与原型的一次性审核存在差异（见差异1）。**

---

### 不一致3：会员状态字段

| 文档 | 状态枚举 |
|------|---------|
| 需求整理_V2 | 审核中 / 已驳回 / 有效会员 / 已过期 |
| 完整规格_V2 §6.1.4 | 未申请 / 审核中 / 已驳回 / 有效会员 / 已过期 |
| 原型 | not_member / pending / rejected / active / expired |

基本一致，但完整规格多了一个"未申请"，原型用 `not_member` 对应。"已驳回"在原型中有独立的 `rejected` 状态，而在完整规格中更倾向于理解为 pending 的一个变体。
