# 单文件离线 HTML 打包流程

## 目标

将 React 19 + Vite 7 + Tailwind CSS 4 的 SPA（中国古生物学会官网，~25 页）打包为一个**独立的 `singlefile.html` 文件**，双击即可在浏览器中用 `file://` 协议打开，不依赖任何服务器、CDN 或网络连接。

---

## 核心挑战

SPA 在 `file://` 协议下运行面临 4 个根本问题：

| # | 问题 | 影响 |
|---|------|------|
| 1 | 路由依赖 History API（`pushState`） | 切换页面或刷新后 404 |
| 2 | 字体从 Google Fonts CDN 加载 | 离线时字体全部丢失，排版错乱 |
| 3 | Google Maps 通过代理加载 JS SDK | 离线时地图组件崩溃或空白 |
| 4 | Vite 默认输出多个文件（JS/CSS 分离） | 无法单文件分发 |

对应的 4 项改造：

| # | 改造 | 方案 |
|---|------|------|
| 1 | History 路由 → Hash 路由 | 编译时常量条件分支 + wouter `useHashLocation` |
| 2 | CDN 字体 → Base64 内联 | 预构建脚本下载 woff2，base64 编码写入 CSS |
| 3 | 地图 → 优雅降级 | `loadError` 状态 + 占位 UI |
| 4 | 多文件 → 单文件 | `vite-plugin-singlefile` 内联所有 JS/CSS/字体 |

## 改造详解

### 1. 条件 Hash 路由

**涉及文件**：`client/src/App.tsx`

**原理**：利用 Vite 的 `define` 在编译时注入常量 `import.meta.env.VITE_HASH_ROUTING`，让 Rollup 的 dead-code elimination 在构建时决定使用哪种路由。

```typescript
// client/src/App.tsx
import { Router, Route, Switch } from "wouter";
import { useHashLocation } from "wouter/use-hash-location";

// 编译时常量：单文件构建时为 "true"，正常构建时为 undefined
const USE_HASH = import.meta.env.VITE_HASH_ROUTING === "true";

function AppRouter() {
  const routes = (
    <Switch>
      {/* 所有 Route 定义完全不变 */}
      <Route path="/" component={SocietyHome} />
      <Route path="/party" component={Home} />
      {/* ... 其余 20+ 条路由 */}
      <Route component={NotFound} />
    </Switch>
  );

  if (USE_HASH) {
    return <Router hook={useHashLocation}>{routes}</Router>;
  }
  return routes;
}
```

**关键点**：

- `Link`、`useLocation` 等组件在两个模式下 API 完全兼容 — **25 个页面组件无需任何修改**
- 正常构建（`pnpm build`）：`USE_HASH` 为 `false`，`useHashLocation` 模块被 tree-shake，产物 1691 个模块
- 单文件构建：`USE_HASH` 为 `true`，包含 hash routing 代码，产物 1692 个模块

**URL 对比**：

```
正常构建:  http://localhost:3000/party
单文件构建: file:///D:/xxx/singlefile.html#/party
```

### 2. 字体 Base64 内联

**涉及文件**：`scripts/download-fonts.ts`、`client/singlefile.html`

**原理**：在 Vite 构建前运行预构建脚本，将 Google Fonts 的 woff2 文件下载、base64 编码、写入本地 CSS 文件，然后 Vite 将该 CSS 作为普通样式内联到最终 HTML 中。

**脚本流程**：

```
┌──────────────────────────────────────────────────┐
│ scripts/download-fonts.ts                        │
├──────────────────────────────────────────────────┤
│ 1. 用 Chrome Windows UA 请求 Google Fonts CSS   │
│    获取 woff2 格式（而非 woff/ttf）             │
│                                                  │
│ 2. 解析 @font-face 块，提取 woff2 URL            │
│    url(https://fonts.gstatic.com/.../xxx.woff2)  │
│                                                  │
│ 3. 逐个下载 woff2 文件（最多重试 3 次）          │
│                                                  │
│ 4. base64 编码                                   │
│    Buffer → data:font/woff2;base64,...           │
│                                                  │
│ 5. 替换 CSS 中的远程 URL → data URI              │
│                                                  │
│ 6. 补充 .material-symbols-outlined 工具类        │
│    （Google Fonts loader 中的默认样式）          │
│                                                  │
│ 7. 输出: client/src/generated/fonts.css          │
└──────────────────────────────────────────────────┘
```

**字体覆盖**：

| 字体 | 用途 | 字重 |
|------|------|------|
| Libre Caslon Text | Logo 中英文标题（`.xingkai-script`） | 400, 700 |
| Hanken Grotesk | 正文、导航、按钮 | 400, 600, 700 |
| Material Symbols Outlined | 图标字体 | Variable (FILL 0~1) |

**两个 HTML 入口的差异**：

```html
<!-- client/index.html (正常构建) -->
<link href="https://fonts.googleapis.com/css2?family=Libre+Caslon+Text..." rel="stylesheet">

<!-- client/singlefile.html (单文件构建) -->
<link rel="stylesheet" href="/src/generated/fonts.css" />
```

输出 CSS 大小：~1829 KB（全部 base64 字体）

### 3. Google Maps 降级

**涉及文件**：`client/src/components/Map.tsx`

**改动点**：

```typescript
// 改动 1：loadMapScript() 错误时 reject Promise 而非只 log
script.onerror = () => {
  script.remove();
  reject(new Error("Failed to load Google Maps script"));  // ← 之前只 console.error
};

// 改动 2：新增 loadError 状态
const [loadError, setLoadError] = useState(false);

// 改动 3：加载失败或 SDK 不可用时显示占位 UI
if (loadError) {
  return (
    <div className="...">
      <span className="material-symbols-outlined">map_off</span>
      <p>地图暂时无法加载</p>
      <p>请连接网络后刷新页面</p>
    </div>
  );
}
```

离线打开文件时，MapView 组件会显示友好的占位提示，而非空白区域或 JS 报错。

### 4. Vite 内联构建

**涉及文件**：`vite.singlefile.config.ts`

**完整配置**：

```typescript
import tailwindcss from "@tailwindcss/vite";
import react from "@vitejs/plugin-react";
import path from "node:path";
import { defineConfig } from "vite";
import { viteSingleFile } from "vite-plugin-singlefile";

const PROJECT_ROOT = import.meta.dirname;

export default defineConfig({
  // 编译时强制注入 hash 路由开关
  define: {
    "import.meta.env.VITE_HASH_ROUTING": JSON.stringify("true"),
  },

  plugins: [
    react(),
    tailwindcss(),
    // ⚠️ viteSingleFile 必须放在插件链最后
    // 它在 HTML 生成后运行，将所有外部引用内联
    viteSingleFile(),
  ],

  resolve: {
    alias: {
      "@": path.resolve(PROJECT_ROOT, "client", "src"),
      "@shared": path.resolve(PROJECT_ROOT, "shared"),
      "@assets": path.resolve(PROJECT_ROOT, "attached_assets"),
    },
  },

  envDir: PROJECT_ROOT,
  root: path.resolve(PROJECT_ROOT, "client"),

  // 不复制 public/ 文件 → 输出目录只有 singlefile.html
  publicDir: false,

  build: {
    outDir: path.resolve(PROJECT_ROOT, "dist/singlefile"),
    emptyOutDir: true,
    assetsInlineLimit: 100 * 1024 * 1024, // 100MB → 内联所有资源
    target: "es2020",
    sourcemap: false,
    rollupOptions: {
      input: path.resolve(PROJECT_ROOT, "client", "singlefile.html"),
    },
  },
});
```

**排除的插件**（与正常构建 `vite.config.ts` 对比）：

| 插件 | 正常构建 | 单文件构建 | 原因 |
|------|---------|-----------|------|
| `react()` | ✅ | ✅ | 必需 |
| `tailwindcss()` | ✅ | ✅ | 必需 |
| `viteSingleFile()` | ❌ | ✅ | 单文件核心 |
| `jsxLocPlugin()` | ✅ | ❌ | 开发调试，产线不需要 |
| `vitePluginManusRuntime()` | ✅ | ❌ | Manus 运行时，产线不需要 |
| debug collector | ✅ | ❌ | 开发调试 |
| storage proxy | ✅ | ❌ | 开发代理 |

## 技术要点备忘

1. **`vite-plugin-singlefile` 必须放在插件数组末尾**，因为它在 `generateBundle` 钩子中处理已生成的 HTML，如果前面还有插件要修改产物，需在其之前执行。

2. **`define` 优于 `.env` 文件**：虽然创建了 `.env.singlefile`，但实际起作用的 hash 路由开关是通过 `define` 注入的。`.env.singlefile` 中的值不会被 Vite 原样替换到客户端代码中，`define` 才会。

3. **`publicDir: false` 很关键**：不加这一行，Vite 会把 `client/public/` 下的文件（`__manus__/`、`version.json` 等）复制到输出目录，导致 `dist/singlefile/` 下出现额外文件。

4. **正常构建完全不受影响**：单文件构建使用独立的配置文件（`vite.singlefile.config.ts`）、独立的 HTML 入口（`singlefile.html`）、独立的输出目录（`dist/singlefile/`），与正常开发和生产构建完全隔离。

5. **`useHashLocation` 的 Link 兼容性**：wouter 的 `Link` 组件在 hash 路由模式下会自动在 `href` 前加 `#`，页面组件中使用 `<Link href="/party">` 无需任何修改。
