## 1. 系统定位与核心用户旅程

### 1.1 一句话定位

面向古生物学者的学术会员管理 + 会议服务 SaaS 平台（原型阶段为纯前端 SPA，React 19 + TypeScript + Tailwind CSS 4）。

### 1.2 核心用户旅程

```
注册账号 → 缴纳统一会员费（两阶段） → 绑定专业分会
         → 选择会议 → 缴纳会议注册费（两阶段）
         → 填写参会信息 + 上传摘要 → 完成报名
```

### 1.3 业务规则总览

| 规则 | 说明 | V2 变更 |
|------|------|---------|
| 唯一会员身份 | 只有一种会员身份——中国古生物学会会员 | 不变 |
| 统一会员费 | 一次缴纳，全部分会通用 | **金额改为后台可配置** |
| 11个专业分会 | 自由绑定/解绑 0~11 个 | 新增到期自动解绑 |
| 会议费独立 | 按每场会议单独缴纳，不支持合并 | 不变 |
| 缴费审核 | — | **一阶段 → 两阶段（凭证初审 + 发票终审）** |
| 发票上传 | — | **选传 → 必传 + 7工作日宽限期** |
| 会员到期后 | — | **无规则 → 分级处理（保留/冻结/删除）** |

## 2. 缴费流程（两阶段审核）

### 2.1 为何需要两阶段？

**业务背景：** 学会财务审计要求"凭证 + 发票"双重验证。
- **凭证**（银行汇款回单）→ 证明"已付款"
- **发票**（电子发票）→ 证明"合法合规"

两者缺一不可。旧流程（一阶段：凭证 + 发票一起上传 → 一次性审核）无法满足财务对两阶段独立核验的要求。

**流程优化思路：** 凭证初审通过后立即解锁参会信息填写，让用户在等待发票终审时准备学术材料，减少等待焦虑。

### 2.2 统一会员费缴纳（两阶段）

```
会员费两阶段审核流程：

  Step 1-2: 确认金额（从配置读取）→ 线下汇款
  Step 3: 上传凭证（JPG/PNG ≤5MB）
    提交 → voucher_submitted → 通知："凭证已提交，财务初审中"

  【阶段一：财务初审】
    ├─ 通过 → invoice_pending
    │       → 设置 invoiceDeadline = 初审通过日 + 7 工作日
    │       → 通知："请于 X 月 X 日前上传电子发票"
    │       → 此时可绑定分会
    └─ 驳回 → voucher_rejected（附原因）
            → 重新上传凭证

  Step 4: 上传发票（初审通过后解锁，JPG/PNG/PDF ≤10MB，必传）
    ├─ 截止日前 → 上传 → OCR 比对 → 提交 → invoice_submitted
    └─ 截止日后 → invoice_overdue → 会员资格锁定（仍可上传）

  【阶段二：财务终审】
    ├─ 通过 → active（有效期1年）
    │       → 通知："会员资格已生效，有效期至 YYYY-MM-DD"
    └─ 驳回 → invoice_rejected（重新上传发票）
```

**关键变化：**
- 原：凭证 + 发票一起上传 → 一次审核 → active
- 新：凭证初审 → 发票终审 → active（两阶段独立审核）

### 2.3 会议注册费缴纳（两阶段）

```
会议费两阶段审核流程：

  前置条件检查（4层）：
    已登录 → 会员有效 → 已绑定该分会 → 未过缴费截止日

  【阶段一：凭证初审】
    Step 1-3: 选择会议 → 选择身份与金额 → 线下汇款
    Step 4: 上传凭证 → voucher_submitted

    初审通过 → invoice_pending
      → 【解锁参会信息填写！】（重大改进）
      → 设置 7 工作日发票截止日
    初审驳回 → voucher_rejected

  【阶段二：发票终审】
    填写参会信息 + 上传发票 → OCR 金额比对
    Step 5: 上传发票（必传）→ 比对通过 → invoice_submitted
    Step 6: 财务终审
      ├─ 通过 → confirmed（报名完成）
      └─ 驳回 → invoice_rejected

  宽限期处理：
    invoice_pending + 超 7 工作日 → invoice_overdue
    → 参会信息锁定（已填数据保留）
    → 仍可上传发票，上传后正常进入终审
```

**关键变化：**
- 原：凭证 + 发票同时上传 → 一次性审核 → approved_unfilled → 填参会信息 → submitted
- 新：凭证初审 → 填参会信息 → 发票终审 → confirmed
- 参会信息填写解锁时机大幅提前（从"终审通过后"改为"凭证初审通过后"）

### 2.4 OCR 模拟校验

发票上传环节包含 OCR 金额比对：

```
用户上传发票 → OCR 识别金额 → 与凭证/会议注册费金额比对
  ├── 一致 → 提交进入终审
  └── 不一致 → toast "发票金额与凭证金额不一致，请检查后重新上传"
```

原型阶段为随机模拟（90% 通过率），实际产品需接入 OCR 服务商。

## 3. 分会绑定与会议参与

### 3.1 分会绑定规则

- **前置条件：** 会员资格有效（`status === "active"`）
- **操作方式：** 浏览 11 个分会列表，点击【绑定】/【解绑】，即时生效，无需审核
- **数量限制：** 0~11 个，无上下限约束

### 3.2 会议可见性规则

**用户仅能看到已绑定分会发布的会议。** 未绑定分会的会议在列表中完全不出现。

实现方式：渲染前按 `boundBranches.includes(conf.branchId)` 过滤。

### 3.3 会员到期后的处理

**新增规则：** 会员到期后所有分会绑定自动解除（`boundBranches = []`）。
续费后不自动恢复绑定——用户需手动重新选择。

### 3.4 学生身份验证

**当前方案（原型阶段）：** 用户自行声明身份，线下会议签到时核验学生证。
未来可增加上传学生证照片的可选步骤。

## 4. 发票必传与宽限期规则

### 4.1 规则概述

| 维度 | V1（旧） | V2（新） |
|------|---------|---------|
| 发票是否必传 | 选传（可选） | **必传** ✅ |
| 上传时机 | 与凭证同步骤 | 凭证初审通过后的独立步骤 |
| 宽限期 | 无 | 7 个工作日 |
| 逾期后果 | 无 | 会员资格/参会信息锁定 |
| 手动延期 | 不支持 | 支持（财务后台操作） |

### 4.2 宽限期计算

```
invoiceDeadline = 凭证初审通过日 + 7 个工作日（跳过周六日）

示例：
  初审通过：周三 → 发票截止日：下周五
  初审通过：周四 → 发票截止日：下周一（跨一个周末）
  初审通过：周五 → 发票截止日：下下周二（跨两个周末）
```

### 4.3 逾期后的处理

| 逾期场景 | 后果 | 补救 |
|---------|------|------|
| 会员费逾期 | 会员资格锁定，无法绑定分会/报名会议 | 仍可上传发票；上传 + 终审通过后自动解锁 |
| 会议费逾期 | 参会信息填写锁定（已填数据保留） | 仍可上传发票；上传 + 终审通过后恢复正常 |

### 4.4 截止日提醒

- 距截止日 ≤ 3 天：自动发提醒通知
- 截止日当天：再次提醒
- 逾期后：逾期通知

### 4.5 兜底手段

1. **手动延期：** 财务为单个订单设置 `invoiceExtendedDeadline`，填写延期理由
2. **手动解锁：** 总管理员可为逾期用户绕过宽限期限制
3. **延期记录：** 每次手动操作记录操作人 + 时间戳 + 理由

## 5. 会员到期后的联动处理

### 5.1 核心原则

**"已锁定的不追溯，未锁定的清理"**

### 5.2 分级处理表

| 会员到期时的会议状态 | 处理方案 | 业务理由 |
|---------------------|---------|---------|
| `confirmed`（已确认） | **保留** | 终审已通过，资格不应被追溯取消 |
| `invoice_submitted`（待终审） | **冻结**（`frozenDueToExpiry = true`） | 流程已完成一半，给用户续费 + 恢复的机会 |
| `voucher_submitted`（待初审） | **删除** | 仅完成凭证上传，损失最小 |
| `voucher_rejected`（凭证驳回） | **删除** | 审核未通过，无保留价值 |
| `invoice_pending`（待传发票） | **删除** | 用户未完成发票上传 |
| `invoice_overdue`（发票逾期） | **删除** | 已逾期未补传 |
| `invoice_rejected`（发票驳回） | **删除** | 审核未通过 |
| `unpaid`（未缴费） | 无影响 | 尚未报名 |

### 5.3 续费后的恢复

```
用户续费 + 发票终审通过后：
  1. 自动检测所有 frozenDueToExpiry = true 的会议
  2. 批量解锁（frozenDueToExpiry = false）
  3. 通知用户 "您被冻结的会议资格已恢复"
  4. 用户手动重新绑定分会
```

**设计考量：** 分会绑定不自动恢复——用户可能在到期期间改变了研究兴趣或分会偏好。

## 6. 系统支撑功能

### 6.1 会员费配置

**需求：** 会员费金额不在代码中硬编码，支撑后台动态配置。

配置维度：
- **按年度：** 每年可设置不同金额
- **按类型：** 普通会员（¥200）/ 学生会员（¥100）/ 单位会员（¥5000）独立定价

原型实现：
```
localStorage key: "paleo_membership_fee_config"
格式: { "standard": 200, "student": 100, "corporate": 5000 }

运行时调用：getMembershipFee("standard") → 200
```

### 6.2 通知系统

所有状态变更均触发对应的站内通知（sonner toast + 顶部铃铛），原型阶段仅站内通知，实际产品需集成邮件通知（共 16 个邮件模板）。

### 6.3 数据持久化

原型阶段使用 localStorage 模拟，所有数据以 `paleo_*` 为前缀存储。实际产品需替换为后端 API + 数据库。

## 7. 状态与术语表

### 7.1 会议注册状态（8 种）

| 状态码 | 中文标签 | 颜色 | 用户可见的含义 |
|--------|---------|------|--------------|
| `unpaid` | 未缴费 | 灰色 | 尚未开始缴费流程 |
| `voucher_submitted` | 凭证初审中 | 黄色 | 已上传转账凭证，等待财务确认到账 |
| `voucher_rejected` | 凭证被驳回 | 红色 | 转账凭证有问题，需重新上传 |
| `invoice_pending` | 待上传发票 | 蓝色 + 倒计时 | 凭证已通过，请于 X 日前上传发票；可填参会信息 |
| `invoice_overdue` | 发票逾期 | 橙色 | 发票未在截止日前上传，请尽快补传；参会信息锁定 |
| `invoice_submitted` | 发票终审中 | 黄色 | 发票已上传（OCR 比对通过），等待财务终审 |
| `invoice_rejected` | 发票被驳回 | 红色 | 发票有问题，需重新上传 |
| `confirmed` | ✓ 已确认 | 绿色 | 终审通过，报名已完成 |

### 7.2 会员状态（9 种）

| 状态码 | 中文标签 | 说明 |
|--------|---------|------|
| `not_member` | 尚未入会 | 已注册但未缴费 |
| `voucher_submitted` | 凭证初审中 | 已提交转账凭证 |
| `voucher_rejected` | 凭证被驳回 | 凭证审核未通过 |
| `invoice_pending` | 待上传发票 | 凭证已通过，待传发票；可绑定分会 |
| `invoice_overdue` | 发票逾期 | 发票上传已超期；会员资格锁定 |
| `invoice_submitted` | 发票终审中 | 发票已提交，待终审 |
| `invoice_rejected` | 发票被驳回 | 发票审核未通过 |
| `active` | 会员资格有效 | 正式会员，有效期 1 年 |
| `expired` | 已过期 | 会员资格已到期 |

### 7.3 关键术语

| 术语 | 定义 |
|------|------|
| 凭证 | 银行汇款/转账的电子回单截图，证明"已付款" |
| 发票 | 学会开具的电子发票，证明"合规" |
| 初审 | 财务审核凭证的阶段（阶段一） |
| 终审 | 财务审核发票的阶段（阶段二） |
| 宽限期 | 凭证初审通过后，上传发票的 7 工作日期限 |
| 工作日 | 周一至周五（不含周末和法定假日） |
| 冻结 | 会员到期后对 `invoice_submitted` 会议资格的处理标记 |
| OCR | 光学字符识别，自动提取发票/凭证上的金额信息 |
| 兜底方案 | 后台手动延期/解锁等管理员操作 |

## 8. 与 V1（原型旧版）的差异对照

### 8.1 流程变化

| 场景 | V1（旧原型） | V2（新） |
|------|------------|---------|
| 会议费审核 | 一次审核（凭证+发票一起） | **两阶段**（凭证初审 → 发票终审） |
| 会员费审核 | 一次审核 | **两阶段**（与会议费对齐） |
| 参会信息填写时机 | 终审通过后 | **凭证初审通过后**（大幅提前） |
| 发票上传 | 选传 | **必传** |
| 宽限期 | 无 | **7 个工作日** |
| 会员到期处理 | 无明确规则 | **分级处理**（保留/冻结/删除） |
| 会员费金额 | 硬编码 ¥200 | **后台可配置** |

### 8.2 状态变化

| V1 状态 | V2 对应 | 说明 |
|---------|---------|------|
| `pending` | `voucher_submitted` | 细化为"仅凭证已传" |
| `approved_unfilled` | 删除，合并到 `invoice_pending` | 参会信息填写权限前移至凭证初审通过 |
| `submitted` | `confirmed` | 语义更明确：终审通过 |
| — | `voucher_rejected` | 新增 |
| — | `invoice_pending` | 新增 |
| — | `invoice_overdue` | 新增 |
| — | `invoice_submitted` | 新增 |
| — | `invoice_rejected` | 新增 |

### 8.3 新增能力

| 能力 | 说明 |
|------|------|
| 凭证初审 | 财务可单独审核转账凭证 |
| 发票终审 | 财务可单独审核电子发票 |
| 发票截止日 | 系统自动计算 + 展示 7 工作日倒计时 |
| 逾期检测 | 系统自动检测 `invoice_pending` → `invoice_overdue` |
| 截止日延期 | 财务可手动延长发票上传截止日 |
| 逾期警告 | 距截止日 3 天自动提醒 |
| OCR 金额比对 | 发票上传时自动识别金额并比对 |
| 会员到期分级 | 按会议状态分级处理（保留/冻结/删除） |
| 资格冻结/解冻 | 会员到期冻结 + 续费后解冻 |
| 费用后台配置 | 会员费按年度 + 类型从配置读取 |
