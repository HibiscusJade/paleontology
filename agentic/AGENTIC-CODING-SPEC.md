# Agentic Coding 工作流规范

> **适用项目**：古生物学研究所党建展示与会员系统  
> **规范版本**：v1.0  
> **更新日期**：2026-06-13

## 一、总体目标

Agentic Coding 是将需求、设计、实现、测试、评审和总结持续沉淀到仓库中的协作方式。所有关键产物统一保存在 `agentic/` 目录，确保后续开发者或 AI Agent 可以快速理解项目上下文、恢复任务进度并追溯决策来源。

## 二、目录约定

```
agentic/
├── 00-mrd/                    # 需求原话、访谈记录、业务反馈、原始材料
├── 00-progress/               # 跨会话任务进度快照
├── 01-prd/                    # 阶段1：需求文档、需求拆解、验收标准
├── 02-design/                 # 阶段2：人工高保真设计输入、AI技术设计、页面/数据设计
├── 03-api-docs/               # 阶段2+：接口文档、请求响应示例、错误码
├── 04-code-changes/           # 阶段3+：代码变更记录
├── 05-unit-test/              # 阶段4：测试用例、测试结果、回归清单
├── 06-code-review/            # 阶段5：代码评审报告、风险问题记录
├── 07-summary/                # 阶段6：交付总结、知识库、后续事项
├── 08-self-evolution/         # Agentic 规范复盘与优化
├── ARCHITECTURE-DECISIONS.md  # 架构决策、业务规则、禁止事项
└── AGENTIC-CODING-SPEC.md     # 本规范文档
```

`agentic/` 仅保存协作文档和过程材料，不应作为业务代码模块加入构建系统。

## 三、子模块协作约定

主项目通过 Git Submodule 管理三个业务子项目：

| 子项目 | 路径 | 职责 |
|---|---|---|
| 后端服务 | `PaleontologicalResearchBack/` | 后端接口、权限、会员、内容、活动、审计 |
| 后台管理前端 | `PaleontologicalResearchAdminFront/` | 管理端页面、菜单权限、运营配置、统计看板 |
| 用户前端 | `PaleontologicalResearchFront/` | 公众展示、会员入口、活动报名和个人服务 |

协作规则：

- 修改业务代码前，先进入对应子模块目录检查 `git status`。
- 子模块内的代码变更在子模块仓库提交和推送。
- 回到主仓库后，提交新的子模块指针和必要的 agentic 文档。
- 主仓库不直接复制业务源码，只维护跨项目文档、协作材料和子模块版本指针。
- 克隆主仓库后使用 `git submodule update --init --recursive` 初始化子项目。

## 四、六阶段流程

```
00 MRD 原话记录 -> 01 PRD 人工确认 -> 02 人工高保真HTML + AI技术设计 -> 03 代码实现 -> 04 测试验证 -> 05 代码评审 -> 06 交付总结
```

### 阶段 00：MRD 原话记录

**输出目录**：`agentic/00-mrd/`

MRD 用于保存用户需求原话和业务方原始输入，是 PRD 的事实来源。AI 可以辅助整理格式、标注来源和归类信息，但不应改写用户原意。

文档建议包含：

- 需求来源
- 记录时间
- 参与角色
- 用户原话
- 访谈纪要
- 业务方反馈
- 原始材料链接或附件说明
- 待确认问题

### 阶段 01：需求解析

**输出目录**：`agentic/01-prd/`

PRD 由人工主导维护，是产品范围、业务规则和验收标准的确认文件。AI 可以基于 MRD 辅助整理结构、补齐待确认问题和统一格式，但不能替人工决定需求范围。若 PRD 对原话做了产品化解释，应保留对应 MRD 链接或引用位置。

文档建议包含：

- 背景和目标
- 用户角色
- 功能范围
- 业务流程
- 验收标准
- 非目标和边界

### 阶段 02：技术设计

**输出目录**：`agentic/02-design/`

本阶段分为两类产物：

- 人工设计输入：人工提供可浏览的高保真 HTML，归档到 `agentic/02-design/references/`，作为 UI 设计、交互路径和视觉还原基准。
- AI 技术设计：AI 基于人工确认的 PRD 和高保真 HTML，输出技术方案、页面结构、接口/数据设计、风险点和变更清单。

文档建议包含：

- 功能概述
- 高保真 HTML 引用
- 页面或接口流程
- 数据模型
- 权限与审计设计
- 影响范围
- 风险和兼容性
- 变更文件清单

### 阶段 03：代码实现

**过程记录目录**：`agentic/04-code-changes/`

每次完成一组有意义的代码变更后，记录：

- 日期
- 变更目标
- 变更文件清单
- 核心实现说明
- 已执行的验证命令
- 未完成事项

### 阶段 04：测试验证

**输出目录**：`agentic/05-unit-test/`

测试材料建议包含：

- 单元测试用例
- 接口测试用例
- 页面验收清单
- 权限与边界用例
- 回归测试结果

### 阶段 05：代码评审

**输出目录**：`agentic/06-code-review/`

评审报告优先记录：

- 阻塞问题
- 业务风险
- 权限和数据安全问题
- 兼容性问题
- 测试缺口
- 修复结论

### 阶段 06：交付总结

**输出目录**：`agentic/07-summary/`

总结建议包含：

- 已交付能力
- 关键设计
- 变更范围
- 验证结果
- 遗留事项
- 后续优化建议

## 五、任务续接规范

跨会话或长任务应在 `agentic/00-progress/` 保存进度快照，命名建议：

```
功能名-progress.md
```

进度快照建议包含：

- 当前目标
- 已完成内容
- 正在处理的问题
- 下一步计划
- 关键文件路径
- 最近一次验证结果

## 六、需求变更处理规范

当用户提出“需求变更”“功能调整”“页面修改”“接口调整”“规则变化”等任务时，AI 必须先读取主项目 `agentic/` 文档，再判断影响范围并执行变更。用户只需要说明“按 agentic 流程处理”并给出具体需求，AI 应自动遵循本节流程。

### 6.1 变更前必读文档

每次需求变更前，AI 至少读取：

- `agentic/AGENTIC-CODING-SPEC.md`
- `agentic/ARCHITECTURE-DECISIONS.md`
- `agentic/01-prd/` 中与需求相关的 PRD
- `agentic/02-design/` 中与页面或交互相关的设计说明
- `agentic/03-api-docs/` 中与接口相关的文档，若目录暂无对应文档则记录为待补充
- 最近的 `agentic/04-code-changes/` 记录，了解近期改动背景

如果用户提供的是原始口头需求、聊天记录或尚未确认的新规则，AI 应先把原话或要点沉淀到 `agentic/00-mrd/`，再按影响范围更新 PRD、设计或接口文档。

### 6.2 影响范围判断

AI 读取文档后，应判断变更影响哪些子项目：

| 影响范围 | 目标子项目 | 判断依据 |
|---|---|---|
| 用户前端 | `PaleontologicalResearchFront/` | 官网展示、会员入口、会议报名、个人中心、移动端页面 |
| 后台管理前端 | `PaleontologicalResearchAdminFront/` | 管理端菜单、审核、统计、导出、运营配置 |
| 后端服务 | `PaleontologicalResearchBack/` | 接口、数据库、鉴权、状态流转、文件上传、审计、业务规则 |
| 主项目文档 | `agentic/`、`README.md` | 需求、设计、接口、测试、评审、总结、子模块指针 |

若需求同时影响多个子项目，应先明确前后端契约，再分仓库实现。若需求与既有 PRD、设计基准或架构决策冲突，AI 必须先指出冲突点，并把最终处理结论写入对应 agentic 文档。

### 6.3 推荐用户输入格式

用户可以用以下格式提交变更：

```md
请按 agentic 流程处理这个需求变更。先读取主项目 agentic 文档，再判断影响范围，更新必要文档，修改代码，验证并提交。

需求变更：
……

业务背景：
……

期望效果：
……

验收标准：
1. ……
2. ……
3. ……
```

如果需求较小，用户也可以只写：

```md
请按 agentic 流程处理：
……
```

AI 在信息足够时应直接执行，不应只停留在方案；信息不足且无法合理判断时，应先列出阻塞问题。

### 6.4 文档更新要求

需求变更完成时，AI 应按实际影响更新以下文档：

- 原始需求或新规则：更新或新增 `agentic/00-mrd/`
- 产品范围、业务流程、验收标准变化：更新或新增 `agentic/01-prd/`
- 页面、交互、数据模型、接口设计变化：更新或新增 `agentic/02-design/`
- 接口路径、请求响应、错误码变化：更新或新增 `agentic/03-api-docs/`
- 代码变更完成：新增 `agentic/04-code-changes/YYYY-MM-DD-变更主题.md`
- 验证或测试完成：更新或新增 `agentic/05-unit-test/`
- 发现风险或评审问题：更新或新增 `agentic/06-code-review/`
- 阶段性交付完成：必要时更新或新增 `agentic/07-summary/`

小变更至少应新增或更新 `agentic/04-code-changes/`，说明变更目标、影响文件、验证命令和遗留事项。

### 6.5 执行与提交要求

AI 执行需求变更时应遵循：

- 修改业务代码前，检查主仓库和相关子模块 `git status`，避免覆盖用户未提交改动。
- 在对应子模块内完成代码修改、验证、提交和推送。
- 回到主仓库提交子模块指针和 agentic 文档更新。
- 验证命令应记录到 `agentic/04-code-changes/` 或 `agentic/05-unit-test/`。
- 若验证无法执行，应明确写明原因、影响和替代检查结果。

## 七、维护分工

| 位置 | 主要维护方 | 说明 |
|---|---|---|
| `agentic/00-mrd/` | 人工为主，AI 辅助 | 人工记录或确认用户原话、访谈纪要、业务方反馈和原始材料；AI 可辅助排版、归类、脱敏，不改写用户原意 |
| `agentic/00-progress/` | AI 为主，人工可改 | AI 在长任务、阶段切换或中断前更新进度，人工可修正真实状态 |
| `agentic/01-prd/` | 人工为主，AI 辅助落文档 | 人工确认需求目标、范围、优先级、业务规则和验收标准；AI 可基于 MRD 整理格式和待确认问题 |
| `agentic/02-design/references/` | 人工提供，AI 归档 | 人工提供高保真 HTML；AI 复制快照、记录来源、日期、哈希和使用说明 |
| `agentic/02-design/` | AI 为主 | AI 基于人工 PRD 和高保真 HTML 生成技术设计、页面结构、数据设计和风险分析 |
| `agentic/03-api-docs/` | AI 为主，人工确认 | AI 根据技术设计和接口实现生成和更新文档，人工确认字段语义、权限边界和前后端契约 |
| `agentic/04-code-changes/` | AI 为主 | AI 在代码生成或修改后同步记录变更点、文件清单、验证命令和遗留事项 |
| `agentic/05-unit-test/` | AI 为主，人工补充 | AI 生成测试用例、回归清单和测试结果，人工补充业务验收场景 |
| `agentic/06-code-review/` | AI 初审，人工终审 | AI 发现代码风险、测试缺口和规范问题，人工负责最终质量判断 |
| `agentic/07-summary/` | AI 起草，人工确认 | AI 汇总交付能力、关键设计、验证结果和遗留事项，人工确认交付口径 |
| `agentic/08-self-evolution/` | 人工与 AI 共同维护 | 共同记录协作复盘、流程优化、提示词或规范升级 |
| `AGENTIC-CODING-SPEC.md` | 人工主导，AI 修改 | 工作流规范由人工决定，AI 负责按决定更新文档 |
| `ARCHITECTURE-DECISIONS.md` | 人工主导，AI 补充 | 架构决策、强制规则、禁止事项由人工确认后沉淀 |

原则：MRD/PRD/高保真 HTML 是人工确认的输入源；AI 从技术设计开始承接主要执行工作，并负责整理、同步、检查和过程记录。

## 八、人工维护格式

人工维护内容统一使用 Markdown；人工设计输入使用可浏览的 HTML 高保真文件。写作要求是：保留事实、少做推测、明确状态、留下待确认问题。若 AI 后续基于人工文档或 HTML 生成设计、代码，必须优先尊重人工输入中的原话、决策、交互和约束。

### 8.1 MRD 格式

适用位置：`agentic/00-mrd/`

```md
# 功能名-MRD

## 基本信息

| 字段 | 内容 |
|---|---|
| 需求来源 | 例如：业务方口头沟通 / 会议 / 微信 / 邮件 / 文档 |
| 记录时间 | YYYY-MM-DD HH:mm |
| 记录人 | 姓名 |
| 参与角色 | 用户 / 业务方 / 产品 / 技术 / 运营 |
| 当前状态 | 草稿 / 待确认 / 已确认 / 已废弃 |

## 用户原话

> 原样记录用户或业务方表达。可以分多段，不要自行产品化改写。

## 背景补充

- 相关业务背景
- 现在线下或旧系统怎么处理
- 为什么现在要做

## 期望结果

- 用户希望最终达到什么效果
- 是否有明确时间点或上线要求

## 约束和边界

- 哪些必须做
- 哪些暂时不做
- 涉及哪些角色、页面、数据或权限

## 待确认问题

- [ ] 问题 1
- [ ] 问题 2

## 原始材料

- 链接或附件说明
```

### 8.3 高保真 HTML 人工提供格式

适用位置：`agentic/02-design/references/`

人工提供的高保真 HTML 应满足：

- 可以直接用浏览器打开。
- 尽量包含主要页面、关键交互和状态展示。
- 文件名带日期和版本号，例如：`2026-06-13-中国古生物学会-高保真原型V2.html`。
- 若新版本改变业务流程，应同步说明变化点，便于 AI 判断是否需要更新 PRD 或 MRD。

AI 归档高保真 HTML 时，需要补充一份设计基准说明：

```md
# 高保真原型设计基准

## 基本信息

| 字段 | 内容 |
|---|---|
| 原始文件 | 桌面或来源路径 |
| 仓库快照 | agentic/02-design/references/xxx.html |
| 文件大小 | bytes |
| SHA-256 | 文件哈希 |
| 定位 | UI 设计 + 交互流程 + 高保真原型 |
| 当前状态 | 开发设计基准 / 已过期 |

## 参考优先级

- 页面布局、视觉、组件、交互路径：以高保真 HTML 为准。
- 字段含义、状态规则、权限和业务计算：以 PRD/技术设计为准。

## 变更记录

| 日期 | 版本 | 说明 |
|---|---|---|
```

新增架构决策、强制规则或禁止事项时，按以下结构追加：

| 字段 | 内容 |
|---|---|
| 日期 | YYYY-MM-DD |

| 状态 | 提议中 / 已确认 / 已废弃 |

#### 决策

为什么这样做，以及放弃了哪些方案。

#### 影响

- 对数据的影响

#### 禁止事项

## 十、质量门禁

进入下一阶段前建议确认：

- MRD 是否保留了关键用户原话和原始上下文
- 需求是否有明确验收标准
- 设计是否覆盖权限、数据、接口、页面和异常场景
- 代码是否符合项目既有结构和命名
- 测试是否覆盖主流程、异常流程和权限边界
- 评审问题是否已关闭或明确记录
- 总结是否能支持后续人员快速接手

## 十一、党建与会员系统特别关注点

- MRD 中的用户原话应尽量原样保留，涉及隐私信息时需脱敏。
- 会员个人信息应最小化展示，敏感字段需脱敏或按权限隔离。
- 党建内容需区分草稿、待发布、已发布、下线等状态。
- 活动报名、审核、签到、取消等状态流转需可追踪。
- 后台操作建议保留操作人、操作时间和操作内容。
- 面向公众展示的内容需关注移动端适配、图片比例和加载性能。
