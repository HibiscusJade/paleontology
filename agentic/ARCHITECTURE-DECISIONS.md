## 一、项目业务边界

本系统服务于古生物学研究所党建展示与会员管理场景，核心边界包括：

- 面向公众或会员的党建内容展示。
- 面向会员的注册、档案、活动、组织关系等服务。
- 面向管理员的内容管理、会员审核、活动管理、统计看板和权限配置。

不在当前边界内的能力，应先在 `agentic/00-mrd/` 记录需求原话，再在 `agentic/01-prd/` 中明确需求和范围后实现。

## 二、核心架构原则

| 原则 | 说明 |
|---|---|
| 展示与管理分离 | 前台展示能力和后台管理能力应保持清晰边界，避免后台字段直接暴露给前台 |
| 权限优先 | 会员信息、审核操作、组织管理等能力必须经过权限校验 |
| 状态可追溯 | 会员审核、活动报名、内容发布等流程必须有明确状态和操作记录 |
| 数据最小化 | 前台和普通会员视角只返回必要字段，敏感信息不默认透出 |
| 原话可追溯 | 用户需求原话、访谈纪要和业务反馈应优先进入 MRD，PRD 只做结构化提炼 |
| 文档随变更更新 | 需求、设计、接口、测试、评审和总结应同步沉淀在 `agentic/` |

## 三、仓库拓扑决策

主仓库 `PaleontologicalResearch` 只保存跨项目说明、agentic 协作文档和子模块指针。业务代码分别放在三个子仓库：

| 子仓库 | 主项目路径 | 职责 |
|---|---|---|
| `PaleontologicalResearchBack` | `PaleontologicalResearchBack/` | 后端服务、权限、会员、内容、活动、审计和接口能力 |
| `PaleontologicalResearchAdminFront` | `PaleontologicalResearchAdminFront/` | 后台管理端页面、菜单权限、运营管理和统计看板 |
| `PaleontologicalResearchFront` | `PaleontologicalResearchFront/` | 用户前端、公众展示、会员自服务入口 |

子模块维护规则：

- 子项目业务代码必须在对应子仓库内提交和推送。
- 主仓库只提交 `.gitmodules`、agentic 文档和子模块提交指针。
- 更新子项目后，应回到主仓库提交新的子模块指针，保证协作者克隆主仓库时能复现同一版本。
- 不在主仓库直接复制子项目业务源码，避免主仓库和子仓库内容分叉。

## 四、强制业务规则

### 4.1 会员隐私

- 手机号、证件号、邮箱等敏感字段不得在非必要场景完整展示。
- 后台导出会员信息应受权限控制，并记录操作审计。
- 普通会员只能访问自己的档案和活动记录。

### 4.2 内容发布

- 党建新闻、通知公告、学习资料、活动风采等内容需支持发布状态。
- 对外展示仅返回已发布且未下线的内容。
- 置顶、推荐、排序等运营字段不得影响基础发布时间逻辑。

### 4.3 活动流程

- 活动需区分草稿、报名中、已截止、进行中、已结束、已取消等状态。
- 报名、审核、签到、取消报名等操作应有明确状态流转。
- 同一会员对同一活动的报名应保证幂等，避免重复报名。

### 4.4 审计与追踪

- 后台关键操作建议记录操作人、操作时间、操作对象和操作内容。
- 删除类操作优先采用逻辑删除，避免直接物理删除造成追溯困难。
- 批量操作必须记录影响范围和执行结果。

## 五、数据设计约定

| 场景 | 建议 |
|---|---|
| 主键 | 使用统一 ID 策略，避免业务字段作为主键 |
| 时间字段 | 保留创建时间、更新时间；发布类内容保留发布时间 |
| 删除字段 | 使用逻辑删除字段表达删除状态 |
| 状态字段 | 使用枚举 code 存储，展示文案由应用层转换 |
| 排序字段 | 使用独立排序字段，不复用主键或发布时间表达运营排序 |

## 六、接口设计约定

- 列表接口必须支持分页。
- 管理端接口和前台接口建议分开命名和鉴权。
- 响应结构应稳定，避免将数据库对象直接作为接口返回。
- 错误信息应能让前端区分权限失败、参数错误、状态冲突和系统异常。

## 七、前端展示约定

- 党建展示页面应保证移动端可读性和图片比例一致性。
- 后台管理页面优先保证信息密度、筛选效率和批量操作效率。
- 会员信息页面应避免一次性展示过多敏感字段。

## 八、Agent 修改代码前检查清单

- 是否已阅读相关 PRD 或技术设计。
- 若需求来自口头沟通，是否已在 `agentic/00-mrd/` 保留用户原话。
- 是否明确本次变更属于前台展示、会员端、管理端还是公共能力。
- 是否涉及会员隐私、权限、状态流转或审计。
- 是否需要同步接口文档、测试用例和代码变更记录。
- 是否存在数据库结构、数据迁移或兼容性风险。
